## גרסה 14 — "דרך מי הגעת אלינו?" (onboarding) — נפרס, ממתין לאימות ידני בפרודקשן
שאלת חובה אחת בהצטרפות, עם אכיפה בשרת. מומש בשלושה חלקים ונדחף כ-Commit/Deploy
יחיד לאחר גיבוי.

### מודל הפרופיל
- שני שדות חדשים בפרופיל: `referralSource` (התשובה) ו-`onboardingCompletedAt`
  (חותמת ההשלמה — נקבעת פעם אחת ואינה מתאפסת בעריכה). פרופיל חדש נוצר עם
  שניהם `null`.

### מסך החובה + עריכה
- משתמש שטרם השלים onboarding רואה מסך חסימה מלא (לא מודאל) עם השאלה "דרך מי
  הגעת אלינו?" (טקסט חופשי, חובה אחרי trim, עד 300 תווים). אחרי שמירה: pending
  עובר ל"ממתין לאישור", ו-Admin נכנס לאפליקציה.
- Admin ממלא פעם אחת (approved אוטומטית אך נדרש כל עוד `onboardingCompletedAt`
  חסר). משתמש pending יכול לערוך את התשובה ("עריכת התשובה" במסך ההמתנה);
  approved רגיל / rejected / disabled חסומים מעריכה. ההכרעה בשרת בלבד.

### שרת ו-API
- `session.access.onboarded` נגזר בשרת מ-`onboardingCompletedAt` (לא נוסף מידע
  אחר ל-session).
- `POST /api/profile { op:'setReferral', text }` — שמירה ע"י המשתמש עצמו
  (userId מהסשן). `GET /api/profile` — מחזיר למשתמש המחובר בלבד את
  `referralSource` ו-`onboardingCompletedAt` (ולא שדות אחרים).
- `setApprovalStatus` חוסם מעבר ל-`approved` ללא `onboardingCompletedAt`
  (reason `onboarding_incomplete`); reject/disable אינם חסומים.

### תצוגת Admin
- מסך ניהול המשתמשים מציג את התשובה ("דרך מי הגיע אלינו"), תווית "טרם השלים
  בקשת הצטרפות" ל-pending לא-שלם (עם הסבר וכפתור "אשר" מוסתר), ו-"לא קיים
  מידע — משתמש ותיק" ל-approved ותיק. משתמשים approved ותיקים אינם מושפעים
  ואינם נחסמים.

### לא כלול (עתידי)
- התראת `user_pending` (N2) ומייל — טרם מומשו.

## גרסה 13 — N1 (מערכת התראות in-app) — נפרס ואומת בפרודקשן
מערכת התראות פנימית (in-app) שנוצרות בשרת בלבד, best-effort ואינן מכשילות
פעולה עסקית. מומשה בארבעה חלקים ונדחפה כ-Commit/Deploy יחיד.

### סוגי ההתראות (N1)
- `request_new` — לבעל האימון (`ownerId`) כשנשלחת בקשת תיאום חדשה.
- `request_accepted` — ליוצר הבקשה (`requesterId`) כשהבקשה מאושרת.
- `request_rejected` — ליוצר הבקשה כשהבקשה נדחית.
- כולן נוצרות רק אחרי הצלחת המוטציה העסקית (CAS), עם `id` דטרמיניסטי
  (`{type}:{requestId}`) למניעת כפילויות. פעולה עצמית (נמען=מבצע) אינה מתריעה;
  פרסום legacy ללא `ownerId` מדלג על `request_new` ונרשם בלוג.

### אחסון (נפרד לכל משתמש)
- Hash ב-Redis לכל משתמש: `notif:{userId}`, field=notifId → אובייקט ההתראה.
- create-if-absent אטומי דרך HSETNX (לא דורס `readAt`/`createdAt` קיימים);
  מצב קריאה נגזר מ-`readAt`; שמירה של עד 50 התראות אחרונות למשתמש.

### API
- `GET /api/notifications` → `{items, unreadCount}` של המשתמש המחובר בלבד
  (מיון מהחדש לישן, `Cache-Control: no-store`).
- `POST /api/notifications` → `markRead` (אידמפוטני) / `markAllRead`.
- אכיפת הרשאות עקבית עם הפרויקט: 403 ללא session/לא-approved; היצירה פנימית
  בלבד (הלקוח לא שולח userId).

### ממשק
- פעמון בסרגל המשתמש (רק ל-approved) עם Badge לא-נקראו ופאנל מהחדש לישן.
- `markRead` בלחיצה (אופטימי) + `markAllRead`; ניווט מההתראה למסך הקיים
  (פרסום/תיאום) בלי Route חדש; מצבי טעינה/ריק/שגיאה; מובייל + RTL + נגישות.
- Polling כל 60 שניות + רענון בחזרה ל-focus (visibilitychange), בלי
  WebSocket/SSE; הגנות מפני מרוצים ו-interval/listener שנשארים אחרי unmount.

### לא כלול (עתידי)
- אין מייל ואין Push. N2 (`request_cancelled`, `training_cancelled`,
  `user_pending`) ופיצ'ר "דרך מי הגעת אלינו?" — טרם מומשו.

## גרסה 12 — שלב 3 (מערכת משתמשים והרשאות) — נפרס ואומת בפרודקשן
מומש בשלבים מקומיים ונדחף כ-Push/Deploy יחיד לאחר הקמת הסביבה וגיבוי.
אומת בפרודקשן: התחברות Google, זיהוי אדמין, משתמש חדש→pending, אישור אדמין,
חסימה לפני אישור, ואכיפת בעלות — כולם עובדים.

### כניסה — Google בלבד ל-MVP
- ספק ההתחברות היחיד הפעיל הוא Google. תשתית ה-Magic Link (Resend) נשמרה
  בקוד אך נטענת רק אם `AUTH_RESEND_KEY` ו-`EMAIL_FROM` מוגדרים; ה-Build
  והפריסה אינם תלויים בהם. מסך ההתחברות מציג כפתור Google בלבד.
- `ADMIN_EMAILS` מגדיר את מנהלי המערכת (נגזר בשרת, לא נשמר בפרופיל).

### תשתית ו-API (חלקים 1–3)
- לקוח Redis משותף (`lib/redis.js`) ועוזרי CAS/collection (`lib/collection.js`):
  קריאה אטומית + `mutateCollectionServer` (read→mutate→CAS בשרת, retry וחוזה
  block מפורש `{block, reason, httpStatus}`).
- הוסר ה-PUT של הקולקציה המלאה (חשיפה קריטית). כל שינוי עובר כ-command
  ממוקד (POST {op,...}); השרת קורא→משנה→CAS על נתונים טריים.

### הרשאות ומשתמשים (חלקים 4a–4b)
- Auth.js (NextAuth v5) עם Google + Magic Link (Resend), `session` מבוסס-DB
  ומתאם Upstash (prefix `auth:`).
- פרופילי משתמש נפרדים (`lib/users.js`, prefix `profile:`) עם
  `approvalStatus`: pending / approved / rejected / disabled; ההחלטה האחרונה
  גוברת, מעברים מבוקרים.
- כללי הרשאה (`lib/authz.js`): Admin נגזר מ-`ADMIN_EMAILS` (לא נשמר); רק
  `approved` מקבל גישה ל-API; בעלות (`ownerId`/`requesterId`) נאכפת בשרת.

### אכיפה בשרת (חלק 4c)
- כל endpoint של נתונים דורש session מאושר, אחרת 403 בלי שינוי נתונים.
- יצירה משייכת בעלות בשרת (`ownerId`/`requesterId` מה-session, לא מהלקוח);
  פעולות בעלים (accept/reject/stage/exec/override/cancel) נבדקות מול הבעלים
  על נתונים טריים בתוך המוטטור.

### ממשק (חלק 4d)
- שער כניסה: מסך התחברות (Google + אימייל), ומסכי מצב חשבון
  (ממתין/נדחה/הושבת) לפי `approvalStatus`.
- מסך "ניהול משתמשים" ל-Admin (אישור/דחייה/השבתה).
- הסתרת פעולות ניהול בממשק ממי שאינו בעל האימון (בנוסף לאכיפה בשרת).

## גרסה 11 — שלב 2A (מודל בקשות ותיאום מרובה)

### 2A.2 — כפתורי פעולה לפי סטטוס בכרטיס הבקשה (commit 0d23e07)
- כפתור "WhatsApp לכוח" מוצג בכרטיס החיצוני רק לבקשה `accepted`.
- כפתור "דחה" מוצג רק לבקשה `pending`.
- ביטול בקשה `accepted` מתבצע רק ממסך פרטי הבקשה ("בטל את בקשת התיאום הזו").
- שינוי UI בלבד; ללא שינוי במודל, בלוגיקה או ב-API.

### 2A.1 — סרגל התקדמות בכרטיס הבקשה + צבע לפי שלב (commit 84a46be)
- סרגל "התקדמות התיאום" הועבר לתוך כרטיס הבקשה שהתקבלה בלבד, ומבוסס על
  ה-`coordinationStatus` של אותה בקשה. `updateCoordinationStage` מעדכן רק
  את הבקשה ואינו כותב ל-postings.
- גזירת סטטוס האימון לפי שלב הבקשה שהתקבלה: "הוחלט על שת״פ" ומעלה → "בוצע
  תיאום" (כחול); "תיאום ראשוני" → "בתהליך תיאום" (כתום).
- הוסר אזור "סטטוס האימון" הידני; ביטול אימון עבר לכפתור נפרד "בטל אימון"
  (אדום) עם "פתיחה מחדש" שמחזירה לגזירה. הוסר הסרגל הכפול ממסך פרטי התיאום.

### 2A — מודל סטטוסים חדש, ריבוי בקשות, accepted יחיד (commit 7238014)
- סטטוס בקשה: ממתינה / התקבלה / נדחתה / בוטלה (`active` ישן ממופה בקריאה
  ל-`pending`, ללא הגירת נתונים).
- סטטוס אימון נגזר בזמן קריאה מהבקשות + override ידני (במקום אחסון וסנכרון).
- אכיפת "בקשה מאושרת אחת לכל אימון" בתוך המוטטור על נתונים טריים (חוזה
  block מפורש; retry opt-in אחד על 409; מדיניות הקונפליקטים הכללית ללא שינוי).
- מפרסם רואה את כל הבקשות עם אישור/דחייה per-request ו-WhatsApp למספר הכוח
  של הבקשה. שפת צבעים אחידה: פנוי=ירוק, בתהליך=כתום, בוצע=כחול, בוטל=אדום.
